## What this sprint tested

Five hard-math problems (one split into three recursion-style variants),
attempted through both the `Solve` action (single theorem body) and the
`SubmitModule` action (helper defs/theorems + root theorem, added in this
same version line). Goal: find where the v0.3.1 mechanism holds up and where
it breaks, and turn every break into either a filed issue or a ruled-out
hypothesis ‚Äî not just a pass/fail count.

### 1 ‚Äî Three-variable AM-GM-type inequality

```lean

theorem root_theorem : ‚àÄ a b c : ‚Ñù, a^2 + b^2 + c^2 ‚â• a*b + b*c + c*a := by

```

### 2 ‚Äî Nesbitt's inequality

```lean
import Mathlib.Tactic.Ring
import Mathlib.Tactic.NormNum
import Mathlib.Data.Real.Basic
import Mathlib.Tactic.Linarith
import Mathlib.Tactic.FieldSimp
import Mathlib.Tactic.Positivity

theorem root_theorem : ‚àÄ a b c : ‚Ñù, 0 < a ‚Üí 0 < b ‚Üí 0 < c ‚Üí a/(b+c) + b/(c+a) + c/(a+b) ‚â• 3/2 := by
intro a b c ha hb hc
have hbc : 0 < b + c := by linarith
have hca : 0 < c + a := by linarith
have hab : 0 < a + b := by linarith
rw [ge_iff_le, div_add_div _ _ hbc.ne' hca.ne', div_add_div _ _ (mul_pos hbc hca).ne' hab.ne', le_div_iff‚ÇÄ (mul_pos (mul_pos hbc hca) hab)]
ring_nf
nlinarith [sq_nonneg (a - b), sq_nonneg (b - c), sq_nonneg (c - a), mul_pos ha hb, mul_pos hb hc, mul_pos hc ha]
```

First attempt failed on `le_div_iff` (renamed to `le_div_iff‚ÇÄ` in the pinned
Mathlib commit) ‚Äî a real API-drift catch, not a ChatDB bug.

### 3 ‚Äî Induction with explicit branches

```lean
import Mathlib.Tactic.Ring
import Mathlib.Tactic.NormNum

theorem root_theorem : ‚àÄ n : ‚Ñï, 0 ‚â§ n := by
intro n
induction n with
| zero =>
    exact Nat.le_refl 0
| succ k ih =>
    exact Nat.le_succ_of_le ih
```

Deliberately stress-tested whether the gateway's blanket line-indent
transform breaks Lean 4's significant-whitespace `induction ... with | ... =>`
branch syntax. It doesn't ‚Äî hypothesis disproven on first attempt.

### 4c ‚Äî Custom recursion: well-founded `log2`

```lean
namespace ChatDB.P_39febe2e13d44d49

def log2 : ‚Ñï ‚Üí ‚Ñï :=
  fun n => if n ‚â§ 1 then 0 else 1 + log2 (n / 2)

theorem log2_eight : log2 8 = 3 := by
  simp [log2]

end ChatDB.P_39febe2e13d44d49
```

First attempt (`decide`) failed: division-based well-founded recursion
compiles to `WellFounded.fix`, which the kernel doesn't reduce through
`decide`/`rfl`. Second attempt (`simp [log2]`, using the auto-generated
equation lemmas) closed it ‚Äî expected Lean behavior, not a bug.

## The one that failed: mutual recursion (issue #19)

`isEven`/`isOdd`, each referencing the other, submitted as two separate
`Def` items ‚Äî rejected with `rejection_diagnostic: "Unknown identifier
isOdd"`. Root cause: `module_items` rendered as a flat sequential list with no
forward references, and Lean's `mutual ... end` block (the only real way to
express this) was entirely banned as a client-suppliable construct, with no
substitute mechanism.

Filed as **[issue #19](https://github.com/Mnehmos/llm-driven-proof-search/issues/19)**
and fixed later the same session: `LeanModuleItem::MutualGroup { members }`
lets a client mark 2+ `Def`/`Theorem` items as a group; the server renders
them together inside one server-owned `mutual ... end` block (client still
never writes `mutual`/`end` itself). Verified by replaying this exact repro
through the harness post-fix ‚Äî `kernel_verified` on the first attempt, and
`proof_export(format="lean")` re-assembles the identical block on replay.
Shipped in `v0.3.1`. See `docs/submit_module.md`'s "Mutual recursion" section.

## Hypotheses tested and ruled out

Two suspected systemic bugs were tested and **disproven** ‚Äî worth recording
as negative results so they aren't re-litigated:

- **Recursion-shape limits in `LeanModuleItem::Def`** ‚Äî suspected that only
  certain recursion shapes would compile inside a bare `Def`'s `body`
  expression. Disproven: course-of-values recursion (4a/4b) and well-founded
  recursion (4c) both work fine via `fun n => match n with | ... => ...`.
- **Indent-transform breaking induction syntax** ‚Äî suspected the gateway's
  blanket line-indent transform would corrupt Lean 4's significant-whitespace
  `induction ... with | pat => ...` branch syntax. Disproven by problem 3.

## Proof-pattern lessons

Reusable doctrine from this sprint (seeded into issue #24's proof-pattern
memory library when that ships):

1. **Mathlib API drift**: a lemma existing under one name in your training
   data may be renamed in the pinned commit (`le_div_iff` ‚Üí `le_div_iff‚ÇÄ`
   here). Try `lean_declaration_lookup` on the exact name before assuming a
   tactic-level fix is needed.
2. **Def-wrapped decidable `Prop`**: `decide` alone can fail on a goal
   wrapped behind a `def` ‚Äî `unfold <name>` first, then `decide`.
3. **Well-founded recursion**: `rfl`/`decide` may not reduce it (expected ‚Äî
   it compiles to `WellFounded.fix`, not a structural fixpoint). Try
   `simp [<def_name>]` to expose the auto-generated equation lemmas instead.
4. **Structural (course-of-values) recursion**: `rfl` closes it directly,
   at shallow and deeper call values alike ‚Äî no special handling needed.
5. **Mutual recursion**: use `LeanModuleItem::MutualGroup`, never two
   forward-referencing bare `Def` items ‚Äî the latter is a hard rejection,
   not a proof-difficulty problem.
6. **Real-number inequalities**: import `Mathlib.Tactic.Linarith` /
   `Mathlib.Tactic.Nlinarith` (not just the base `Ring`+`NormNum` manifest)
   and `Mathlib.Data.Real.Basic` through `problem_imports` up front.
7. **`unsafe_dev_attestation=true`**: correct for fast play-testing iteration,
   but every episode created under it is permanently capped at
   `kernel_verified` and excluded from default dataset exports ‚Äî don't reuse
   attested problem_versions for anything meant to reach `certified`.

rule, duplicate-name detection across a group, injection hardening inside a

reusable dev tool for any future sprint like this one.
